Analysis

Linearity confirmed. Both measured and malioc series fit a + b·kv with R²≈1.000. This is structurally expected: QKT ([1,hd]×[hd,kv]) and S·V ([1,kv]×[kv,hd]) each operate per-element over kv, making total cost linear in kv regardless of implementation.

Bound pipeline consistent. malioc reports load_store as the bound pipeline at all three kv sizes (longest-path cycles: 4610 / 18400 / 36900, scaling exactly 4× and 8× from kv=128). This is consistent with the hardware roofline finding that attention operates below the 20.1 GFLOP/s compute ceiling — the kernel is bandwidth-limited, not compute-limited.

Slope magnitude gap (11.65×) is explained. malioc is a per-thread static cycle counter. It does not model: (1) kernel launch and driver overhead, which dominates the measured intercept a=27.7 µs; (2) the serial softmax bottleneck — work-item 0 executes all KV_LEN iterations while HEAD_DIM−1 threads idle at the barrier, multiplying effective wall-clock time; (3) LPDDR4 bus contention shared with CPU and CIM tile. The ratio converges from 17.4× (kv=128, intercept-dominated) toward ~12× (kv=1024, slope-dominated), which is the expected behaviour.

Validation scope. Static analysis validates the computational structure of the GPU timing model — correct complexity class, correct bound resource, correct scaling behaviour. Absolute latency calibration comes from hardware measurements (m4_gpu.json, n=3 points, max error 1.1%). Together these constitute structural validation of the simulator's GPU timing layer. Full cycle-accurate re-validation would require board access or a licensed Mali RTL model, neither of which is currently available.
