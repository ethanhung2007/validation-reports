CIM Compute Core

The protagonist compute engine of the heterogeneous SoC—all weight-stationary GEMMs (Q/K/V/O projections, FFN gate/up/down, lm_head) live here. This page integrates four layers of results: decode calibration, Card revalidation, prefill amortization, and multi-tile residency cliff.

calibratedfittedsimulatedassumptionborrowed
§0Role of This Chapter

M1 simulates the two most fundamental operators of the CIM compute core. Phase 1 only gets the latency of these two things right, without touching scheduling or multi-unit contention:

① GEMV (decode, M=1)—decode produces one token per step, a weight-stationary matrix × vector. ② GEMM (prefill, M>1)—prefill consumes a batch of M tokens at once, a matrix × matrix, where weight is loaded once and reused by M tokens. All Q/K/V/O projections, FFN gate/up/down, and lm_head in the model fall into these two categories.

On the hardware side, the Metis AIPU is a quad-core, each core with a 512×512 INT8 digital in-memory (D-IMC) crossbar (ISSCC 2024). "Weight-stationary" = weights are written into the crossbar and stay put, activations flow through to perform MACs. Throughput is measured in INT8 GOP/s (not GFLOP/s). M1 is one of the few units that touches real silicon and is Card-revalidated, with nearly all numbers coming from our own on-board measurements.

§1Measurementsilicon · GEMV + GEMM

Measurement protocol: the same 800 MHz AIPU, first measured on Aetina Metis Alpha (pre-production) with run_metis_cim.py + axrunmodel dev_fps, then revalidated on the production Metis Card (16 GiB on-card LPDDR4x) by compiling the same 1×1-conv matmul proxy with axcompile. Both boards run at the same frequency, so GOP/s is directly comparable without clock normalization. We only measure M1's two fundamental operators:

1a · GEMV (decode, M=1): Measure "Effective Throughput G_eff"

Decode is matrix × vector for one token per step. The measurement target is effective throughput G_eff—the actual INT8 GOP/s achieved by a tile of shape (N, K) (N = output channels, K = contraction dimension / input dimension). 13 native single-tile shapes were measured on Alpha (K∈{2048,3072,3584,4096}, N∈{64…2048}), and Card revalidated the same 13 shapes. §2Fitequation · parameter-by-parameter explanation

GEMV (M=1): G_eff 2D Closed Form

W = n_cores × 512 # effective output width dev_lat(M=1, K, N) = Σ_tiles 2·K·n_tile / G_eff(n_tile, K) [µs] G_eff(N, K) = Gmax · N/(N + Na) · K/(K + Kb) [GOP/s]

How the latency formula is derived: Tiles are cut along the output dimension N. Each tile's workload is 2·K·n_tile INT8 ops (factor 2 = one MAC counts as multiply + add, i.e., two ops), divided by that tile's own effective throughput G_eff to get time, then summed. The key point is "each tile is charged by its own width"—the last partial tile is not charged as a full tile.

GEMM (M>1): M-amortization Affine Fit

tile_lat(M) = a + b·M # latency of one 2048×2048 tile full_GEMM_lat(M, K, N) = (K·N / W²) · tile_lat(M) # fractional area, not ceil

Weight is loaded once and reused by M activation columns, so latency vs M is affine (a straight line); its measurement points and fit line are shown in §1b Fig M1·2 above. M=1 decode is a different regime and not included in this fit (mistakenly using G_eff to linearly extrapolate decode to large M would severely overestimate by one to two orders of magnitude; the affine form eliminates this bias). §3Simulation · Multi-tile / Extrapolationno heavy-sim · analytic extrapolation

M1 does not carry any heavy-sim engine (decode is entirely memory-bound, compute ceiling does not need modeling, issue #16). "Simulation" in this section = analytic extrapolation beyond the measurement envelope, supported by a residency-cliff model calibrated from Card-native multi-tile measurements.

Revised previous assumption: axcompile can compile more than just a single 2048×2048 tile. Phase 1.5 compiled natively on Card up to K·N≈17M. M=1 throughput rises smoothly with K·N until knee≈8.2M params (weights resident in on-chip SRAM), then collapses ~3.5× to a DRAM-spill floor of ~70 GOP/s. Real decode FFN/lm_head GEMVs (K·N≥16M) fall in the spill regime—the old tile-sum underestimated by ~3×; the new cliff model is needed for per-op M1 latency to be correct for Phase 2.
